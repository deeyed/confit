# Coding And Documentation Rules

Confit 구현팀은 Parus/Delos의 기본 코드·문서 규칙을 따라야 한다. Confit은 host tool이지만,
Parus/Delos의 build authority가 될 예정이므로 더 느슨한 실험 스크립트처럼 작성하면 안 된다.

## Language

- Core implementation은 C로 작성한다.
- 초기 host portability를 위해 C17 subset을 사용할 수 있으나, Parus/Delos 방향과 맞게 C23-friendly
  code를 유지한다.
- C++와 Rust는 명시 결정 전까지 사용하지 않는다.
- Python은 prototype generator로 쓰지 않는다. Confit의 목표 중 하나는 Python-heavy config tool을
  피하는 것이다.

## Dependency Policy

- TOML parser는 permissive license C library만 고려한다.
- Parser는 adapter 뒤에 숨긴다.
- TUI library를 도입하더라도 core evaluator가 TUI에 의존하면 안 된다.
- Build graph integration은 CMake/QStar가 담당하고, 초기 Confit은 generated `config.h`와 report를 만든다.
- CMake/QStar generated fragment는 core/TUI/Parus/Delos 적용이 안정화된 뒤 도입한다.
- Hosted OS 기능은 `src/host/` 아래에 격리한다.

## Memory And Error Handling

Confit은 host tool이므로 heap 사용이 가능하다. 그러나 API boundary는 명확해야 한다.

- Core model allocation ownership을 문서화한다.
- Error는 작은 numeric code와 diagnostic record를 함께 제공한다.
- 단순 문자열 error를 모든 함수가 들고 다니는 방식은 피한다.
- CLI layer만 사람이 읽는 error message를 format한다.

권장 형태:

```c
typedef enum ConfitStatus {
  CONFIT_OK = 0,
  CONFIT_ERR_INVALID_ARGUMENT = 1,
  CONFIT_ERR_PARSE = 2,
  CONFIT_ERR_SCHEMA = 3,
  CONFIT_ERR_CONFLICT = 4,
} ConfitStatus;
```

## Doxygen Rule

공개 header의 함수, 타입, enum, macro는 한국어 Doxygen 주석을 가져야 한다.

예:

```c
/**
 * @brief Confit option graph를 검증한다.
 *
 * @param graph schema loader가 구성한 option graph.
 * @param report conflict와 dependency 오류를 기록할 report.
 * @return 검증에 성공하면 CONFIT_OK, 실패하면 오류 status.
 */
ConfitStatus confit_graph_validate(const ConfitGraph *graph, ConfitReport *report);
```

두 줄 이상의 내부 설명, graph invariant, memory ownership, parser lifetime, generated artifact 계약도
Doxygen block으로 작성한다.

## File Layout

구현이 시작되면 권장 layout은 다음과 같다.

```text
tools/confit/

```

문서만 있는 현재 단계에서는 `include/`, `src/`, `tests/`를 만들지 않는다. 구현팀이 시작할 때 만든다.

## Host Isolation

Confit은 모든 주요 OS에서 동작해야 한다. 따라서 hosted 기능은 core에 직접 새지 않게 한다.

`src/core/`, `src/schema/`, `src/graph/`, `src/resolver/`, `src/generator/`는 다음 API를 직접 호출하지
않는다.

- `fopen`
- `getenv`
- terminal control sequence
- path separator 직접 분기
- OS-specific clock
- locale-dependent formatting

이 기능은 `src/host/` adapter가 담당한다. CLI와 TUI는 host adapter를 통해 file, path, stdout,
terminal capability를 사용한다.

## Generated Code Comments

Generated `config.h`는 사람이 읽을 수 있는 header comment를 가져야 한다.

```c
/*
 * Generated by Confit.
 * Source profile: config/profiles/sim-dsh.toml
 * Do not edit by hand.
 */
```

Generated file에는 timestamp를 넣지 않는다. Reproducibility를 해친다.

## Documentation Policy

Confit 문서는 한국어를 기본으로 한다.

정본에 적합한 내용:

- 영속적인 schema 규칙
- dependency graph 의미
- generated artifact 의미
- runtime boundary
- TUI와 core evaluator의 권한 관계

정본에 부적합한 내용:

- 특정 라운드의 진행 상태
- 임시 workaround
- 로컬 build 실패 기록
- 특정 commit 상태

상태성 기록이 필요하면 별도 log 문서를 만든다.

## Test Policy

초기 구현팀은 다음 test를 최소로 만들어야 한다.

- TOML parse fixture.
- duplicate option id reject.
- unknown dependency reject.
- cycle reject.
- `requires` failure explanation.
- `conflicts` failure explanation.
- `forces` lint.
- profile merge order.
- generated `config.h` golden test.
- generated JSON report golden test.
- generated graph/input manifest golden test.
- Parus/Delos compatibility failure test.

CMake/QStar generated fragment golden test는 해당 generator를 도입하는 시점에 추가한다.

## Release Boundary

Confit은 release profile을 검증하는 도구이므로, Confit 자체가 release image에 들어가면 안 된다.

Delos release profile 예:

```text
DELOS_CONFIG_DEBUG_DDC=0
DELOS_CONFIG_DEBUG_DSH=0
DELOS_CONFIG_DEBUG_DSH_RX=0
```

Build graph는 이 값을 보고 DDC/DSH source를 제외해야 한다.

## Agent Handoff Rule

다른 Codex 팀이 Confit 구현을 시작할 때는 이 순서로 읽는다.

1. `tools/confit/README.md`
2. `tools/confit/architecture.md`
3. `tools/confit/toml-schema.md`
4. `tools/confit/resolution-dag.md`
5. `tools/confit/generators.md`
6. `tools/confit/cli-tui.md`
7. `tools/confit/coding-and-doc-rules.md`

이 문서를 읽은 뒤에도 애매한 점이 있으면 code를 쓰기 전에 schema와 generated artifact 계약부터
질문해야 한다.
